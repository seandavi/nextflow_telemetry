# Self-host on onclappc02

Production compose for the telemetry API + frontend, served via the
existing Traefik on `cancerdatasci.org`. The Postgres backend lives in
the shared `pg_main` cluster (see
`monode/infrastructure/compose/pg_main_stack/`), one database per
app — this app owns the `nf_telemetry` database.

This is the live production stack: external
`https://nf-telemetry.cancerdatasci.org` resolves to onclappc02
(`140.226.4.71`) with a Let's Encrypt cert issued by Traefik.

## Files

- `docker-compose.yml` — API + frontend, both labelled for Traefik
- `Dockerfile.frontend` — multi-stage build (node → nginx:alpine)
- `nginx.conf` — SPA config with sane caching headers
- `fetch-secrets.sh` — materialize OAuth/session secrets from GCP SM → `.env.secrets`
- `.env.example` — fill in and copy to `.env` (DB connection + hostnames)
- `.env.secrets` — generated by `fetch-secrets.sh`, gitignored, mode 600

## Deploying a code change (routine)

From a checkout on onclappc02:

```sh
git checkout main && git pull          # build context is the working tree — pull first
just deploy-onclappc02
```

`just deploy-onclappc02` runs three steps (see the justfile):

1. `deploy/onclappc02/fetch-secrets.sh` — refresh `.env.secrets` from GCP SM
2. `docker compose build nf_telemetry_api` — rebuild the image from the working tree
3. `docker compose up -d nf_telemetry_api` — recycle the container (~30s of 503; clients retry)

**Migrations are not run by this target.** If the change adds one, run it
from the host first (pg_main only resolves inside docker, so use
`127.0.0.1` from the host):

```sh
set -a; source deploy/onclappc02/.env; set +a
SQLALCHEMY_URI="${SQLALCHEMY_URI/@pg_main:/@127.0.0.1:}" uv run alembic upgrade head
```

Verify after deploy:

```sh
docker ps --filter name=nf_telemetry_api          # healthy?
docker compose logs -f nf_telemetry_api           # clean startup
curl -sS https://nf-telemetry.cancerdatasci.org/health   # 200
```

## Backend topology (current)

```
pg_main                127.0.0.1:5432  → /data/postgres_ducklake/  (shared cluster, db nf_telemetry)
nf_telemetry_api       proxy + pg_main_stack_default  → pg_main:5432/nf_telemetry
nf_telemetry_frontend  proxy net       → built with VITE_API_URL=https://nf-telemetry.cancerdatasci.org
traefik                host net        → routes both hostnames by Docker labels
```

- `nf-telemetry.cancerdatasci.org` → `A 140.226.4.71` (DNS-only / grey cloud).
- `cmgd.cancerdatasci.org` → **Cloudflare-proxied** (orange cloud) as of
  this writing — it resolves to Cloudflare edge IPs, not `140.226.4.71`
  directly. (This diverges from the original grey-cloud cutover plan
  below; if you re-point it, decide grey vs orange deliberately.)

The `pg_main` host only resolves from inside the `pg_main_stack_default`
docker network (which the API container joins). pg_main binds
`127.0.0.1:5432` on the host, so anything run **from the host** (e.g.
migrations) must use `@127.0.0.1:5432` rather than `@pg_main`.

## Secrets

Source of truth is **GCP Secret Manager** (project `cdsci-infra`).
Never invent a parallel copy in a config file, a chat thread, or a
password-manager note. Two materialized runtime artefacts, both
gitignored and derived (not sources of truth):

- `.env` — DB connection (`SQLALCHEMY_URI`) + hostnames (`API_HOST`,
  `FRONTEND_HOST`). The DB password is substituted in from SM by hand.
- `.env.secrets` — OAuth + session secrets, generated by
  `fetch-secrets.sh` and read by compose via `env_file:`.

| Secret (GCP SM) | Role / use | Consumed by |
|---|---|---|
| `cdsci-postgres-admin-password` | `postgres` superuser on `pg_main` | cluster admins; not used by app |
| `cdsci-nf-telemetry-db-password` | `nf_telemetry` login role | `.env` `SQLALCHEMY_URI` |
| `cancerdatasci-oauth-client-id` | Google OAuth client id | `.env.secrets` `OAUTH_CLIENT_ID` |
| `cancerdatasci-oauth-client-secret` | Google OAuth client secret | `.env.secrets` `OAUTH_CLIENT_SECRET` |
| `cmgd-api-session-secret` | session signing key | `.env.secrets` `SESSION_SECRET` |

**Naming convention** is not uniform (history): shared-cluster / DB
secrets use `cdsci-*` (and `cdsci-<app>-db-password` for per-app roles);
the OAuth/session secrets predate that and use `cancerdatasci-*` /
`cmgd-*`. Reconcile if you rotate them.

**Refreshing OAuth/session secrets** (after a rotation in SM):

```sh
cd deploy/onclappc02 && ./fetch-secrets.sh
docker compose up -d nf_telemetry_api      # container reads env at startup only
```

**Rotating the DB password** (~30s of 503 during recreate):

```sh
# 1. Generate + write new version to SM
python3 -c "import secrets, string; print(''.join(secrets.choice(string.ascii_letters+string.digits) for _ in range(32)))" \
  | gcloud secrets versions add cdsci-nf-telemetry-db-password \
      --project=cdsci-infra --data-file=-

# 2. Apply to the Postgres role (psql heredoc; never pass via -c with :'var'):
PW=$(gcloud secrets versions access latest \
  --secret=cdsci-nf-telemetry-db-password --project=cdsci-infra)
PW_ESC=$(printf %s "$PW" | sed "s/'/''/g")
docker exec -i pg_main psql -U postgres -v ON_ERROR_STOP=1 <<SQL
ALTER ROLE nf_telemetry WITH PASSWORD '$PW_ESC';
SQL

# 3. Rewrite deploy/onclappc02/.env with the new $PW, then:
cd deploy/onclappc02 && docker compose up -d --force-recreate nf_telemetry_api
```

If the API can't reach the DB (503 from `/health`), first check whether
`.env` is in sync with the latest secret version.

## First-time setup (already done; recorded for reproducibility)

1. **Database + role** in the shared cluster. Password generated and
   stored in SM — not chosen ad-hoc:
   ```sh
   python3 -c "import secrets, string; print(''.join(secrets.choice(string.ascii_letters+string.digits) for _ in range(32)))" \
     | gcloud secrets create cdsci-nf-telemetry-db-password \
         --project=cdsci-infra --replication-policy=automatic --data-file=-

   PW=$(gcloud secrets versions access latest \
     --secret=cdsci-nf-telemetry-db-password --project=cdsci-infra)
   PW_ESC=$(printf %s "$PW" | sed "s/'/''/g")
   docker exec -i pg_main psql -U postgres -v ON_ERROR_STOP=1 <<SQL
   CREATE ROLE nf_telemetry WITH LOGIN PASSWORD '$PW_ESC';
   CREATE DATABASE nf_telemetry OWNER nf_telemetry;
   SQL
   ```
   (Do **not** `CREATE EXTENSION pg_duckdb` in this DB — the app uses
   plain Postgres; pg_duckdb hooks would only get in the way.)

2. **`.env`** — copy `.env.example` to `.env`, fetch the password from SM,
   substitute into `SQLALCHEMY_URI`. Shape:
   ```
   postgresql+asyncpg://nf_telemetry:<pw-from-sm>@pg_main:5432/nf_telemetry
   ```

3. **Apply migrations** from the host (see the host/127.0.0.1 note above).

4. **Fetch OAuth + session secrets** → `.env.secrets`:
   ```sh
   cd deploy/onclappc02 && ./fetch-secrets.sh
   ```

5. **Build + bring up**:
   ```sh
   cd deploy/onclappc02 && docker compose build && docker compose up -d
   ```
   (For subsequent code changes use `just deploy-onclappc02` — see above.)

## Cutover history (completed May 2026)

The stack was migrated off Cloud Run + Cloud SQL onto onclappc02. The DNS
flip in Cloudflare (`nf-telemetry.cancerdatasci.org` →
`A 140.226.4.71`, DNS-only) is done and Traefik holds a valid Let's
Encrypt cert. Recorded for reference:

- During the flip, Traefik's TLS-ALPN ACME challenge failed (403) while
  DNS still pointed at Cloud Run, which can trip the Let's Encrypt
  ~1-hour rate limit after 5 failures. It self-resolved once DNS flipped.
- Production data bootstrap registered workflows via `POST /api/workflows`
  and seeded samples, then `POST /api/admin/reconcile-jobs`. **Note:** the
  registration body has **no `profile` field** — profile is per-cluster and
  lives in the nf-client config, not the workflow registry. Current shape:
  ```sh
  curl -X POST https://nf-telemetry.cancerdatasci.org/api/workflows \
    -H 'Content-Type: application/json' \
    -d '{
      "workflow_id": "cmgd_2.0.0",
      "version": "2.0.0",
      "repository_url": "https://github.com/seandavi/curatedMetagenomicsNextflow",
      "revision": "2.0.0",
      "max_retries": 1,
      "description": "Runs standard biobakery metaphlan, kraken2/braken, AMR"
    }'
  ```

## Operational notes

- **Routine deploy**: `just deploy-onclappc02` (see top). For a one-off
  manual rebuild: `docker compose build nf_telemetry_api && docker compose
  up -d nf_telemetry_api`.
- **Migrations** run from the host, not the API container
  (`uv run alembic upgrade head` with the `@127.0.0.1` override).
- **Frontend updates** are baked at build time; rebuild only if the API
  URL changes or the bundle is stale.
- **Backups** are not configured at this layer. Tracked in issue #84
  (pg_basebackup + WAL archiving at the shared cluster level).
- **Logs**: `docker compose logs -f nf_telemetry_api` for app logs;
  `docker logs traefik` for routing + cert issues.
